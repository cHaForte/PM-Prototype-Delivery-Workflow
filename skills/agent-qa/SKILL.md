# agent-qa — 质量保证智能体

## 模式 A：质量检查
检查维度：功能完整性 / 代码质量 / 用户体验 / 响应式适配
问题分级：P0（阻塞）~ P3（优化建议）

---

## 模式 B：测试大纲撰写

### 输入
1. 需求变更单：./vault/01-需求管理\需求变更单\[项目名]-需求变更单.md
   → **测试大纲必须以此为基准，逐项覆盖需求中的功能指标**
2. 最终原型：./prototypes/[项目名]\

### 核心原则
测试大纲必须参考需求变更单进行产出。需求变更单中的每个功能需求、每个交互流程，都必须有对应的测试用例覆盖。

---

### 测试用例编号规则（重要）

每个功能模块有**独立的唯一前缀**，用例编号格式：`[模块前缀]-[序号]`

**前缀生成规则：** 从模块名称中提取 2~4 个拼音首字母，要求直观可识别。

| 模块名称示例 | 前缀 | 用例编号示例 |
|:-----------|:----:|:-----------:|
| 上传模块 | SCMK | SCMK-001, SCMK-002 |
| 详情页模块 | XQY | XQY-001, XQY-002 |
| 全文阅读页 | QWYD | QWYD-001, QWYD-002 |
| 我的标注页 | WDBZ | WDBZ-001, WDBZ-002 |
| 历史记录 | LSJL | LSJL-001, LSJL-002 |
| 精选模块 | JX | JX-001, JX-002 |
| 最近常听 | ZJCT | ZJCT-001, ZJCT-002 |
| 排行榜 | PHB | PHB-001, PHB-002 |
| 播放组件 | BFZJ | BFZJ-001, BFZJ-002 |
| 首页 | SY | SY-001, SY-002 |
| 检索 | JS | JS-001, JS-002 |

**序号规则：**
- 每个模块独立计数，从 001 开始
- 同一模块内按功能点的自然顺序递增

## 4 测试用例

### 4.1 [模块名称] 测试用例

每个功能点对应一个或多个测试用例。每个用例格式如下：

**用例 [模块前缀]-[序号]：[用例名称]**

- **测试目的：** [验证该功能的什么特性]
- **需求覆盖：** [对应 3.1 中的功能点]
- **优先级：** [核心 / 重要 / 一般 / 较低]
- **前置条件：** [进入该测试需要的前提]
- **测试步骤：**
  1. [具体操作步骤]
  2. [具体操作步骤]
- **预期结果：** [执行后的预期表现]
- **结果评价：** 与预期结果一致

**优先级规则：**
- 核心：主流程、核心功能
- 重要：分支流程、关键交互
- 一般：次要功能、边界条件
- 较低：UI 细节、优化项

---

## 5 结果评价

1. 测试用例的覆盖度不能低于80%
2. 核心/重要测试用例必须执行
3. 测试结果合格率不能低于95%

### Word 文档生成（必须执行）

Markdown 文件保存后，必须运行以下命令生成对应的 Word 文档：

命令格式：
`
python ./scripts/generate-docx.py "[MD文件路径]" "[DOCX文件路径]"
`

执行确认：
- 如果命令执行成功，输出 "已生成: [路径].docx"
- 如果执行失败，重试一次
- 在回复中展示 Word 文件的完整路径

命令示例：
`
python ./scripts/generate-docx.py "./vault/01-需求管理\需求变更单\项目名-需求变更单.md" "./vault/01-需求管理\需求变更单\项目名-需求变更单.docx"
`


### 编码规则
所有输出文件必须使用 UTF-8 编码。
