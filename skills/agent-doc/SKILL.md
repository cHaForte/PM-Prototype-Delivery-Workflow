# agent-doc — 文档管理智能体

## 模式 A：交付打包
收集所有阶段产出物，生成变更记录和交付清单。

---

## 模式 B：用户帮助文档撰写

### 核心写作原则
1. **用户视角**：以"您"称呼用户，从用户的使用场景和痛点出发
2. **先价值后操作**：先告诉用户这个功能解决什么问题，再写具体怎么做
3. **口语化但不随意**：专业、清晰，避免技术术语堆砌
4. **图文结合**：用「图X」标记截图位置

### 输入
- 最终原型：./prototypes/[项目名]\
- 需求整理摘要：./vault/01-需求管理\需求整理摘要\[项目名]-需求整理摘要.md

---

### 产出结构（严格按此顺序）

## # [项目名] 用户帮助文档

### 第一部分：痛点引入 + 产品概述

标题可以写成问句或痛点句，例如：
「您还在为 [用户痛点] 而苦恼吗？」

然后说明：
1. 这个功能/产品解决什么问题
2. 核心价值是什么
3. 适用场景有哪些

> 示例（来自模拟评审帮助文档）：
> 您还在为如何提升各类稿件的质量而苦恼吗？
> 申请书无从下手：国自然申请书逻辑不清，重点不突出...
> 现在，告别瓶颈，迎接突破！[产品名] 是一款基于人工智能技术的...
> [产品名]支持对多种类型的...进行评审，包括但不限于...

---

### 第二部分：核心功能与亮点

每个功能点按以下格式描述：

**功能名称：一句话说明**

[详细说明]：这个功能能做什么，用户使用它能获得什么好处。

> 示例：
> 1. 智能化匹配专家：告别盲审，精准对接领域专家！
> 无论是科研基金申请还是学位论文撰写，[产品名]都能为您提供精准的专家支持...

功能数量建议：
- 简单项目：2~3 个功能点
- 复杂项目：3~5 个功能点

---

### 第三部分：操作指南

从用户视角，按操作顺序分步骤编写。每步包含：
1. **入口说明**：用户在哪里能找到这个功能
2. **操作步骤**：按序号列出每一步操作
3. **截图标记**：在关键位置用「图1 说明文字」标注

格式示例：

**从 [入口位置] 进入 [功能名称]**

从 [导航/菜单/按钮位置] 找到 [功能入口]，点击即可进入 [功能页面]。

**1. [操作步骤名称]**

点击 [具体按钮]，打开 [弹窗/页面]。在弹窗中：
- [操作A]：[说明]
- [操作B]：[说明]

图1 操作界面说明

**2. [下一步操作名称]**

[操作说明]

图2 [截图说明]

**3. [再下一步操作]**

...

> 注意：每个「图X」代表一个截图占位，后续替换为实际截图。
> 截图编号从"图1"开始连续编号。

---

### 第四部分：常见问题

3~5 个用户最可能遇到的问题，格式：

**Q：[问题]**
A：[解答]

问题类型建议：
- 操作类（如"如何开始"）
- 限制类（如"文件大小限制"）
- 异常类（如"出错怎么办"）
- 费用类（如"怎么收费"）

---

### 写作风格要求

| 要素 | 要求 |
|:----|:----|
| 人称 | 第二人称"您"，不出现"用户" |
| 语气 | 亲切、自信、专业 |
| 句式 | 短句为主，避免超过 40 字的句子 |
| 技术术语 | 涉及时需通俗解释 |
| 截图 | 用「图X」标记，截图后替换 |

---

### 保存路径

MD：./vault/04-文档管理\用户帮助文档\[项目名]-用户帮助文档.md


### Word 文档生成（必须执行）

Markdown 文件保存后，必须运行以下命令生成对应的 Word 文档：

命令格式：
`
python ./scripts/generate-docx.py "[MD文件路径]" "[DOCX文件路径]"
`

执行确认：
- 如果命令执行成功，输出 "已生成: [路径].docx"
- 如果执行失败，重试一次
- 在回复中展示 Word 文件的完整路径

命令示例：
`
python ./scripts/generate-docx.py "./vault/01-需求管理\需求变更单\项目名-需求变更单.md" "./vault/01-需求管理\需求变更单\项目名-需求变更单.docx"
`


### 编码规则
所有输出文件必须使用 UTF-8 编码。
